## 1. 전략 변경 배경

### 변경 사항
- ❌ **이전**: 사전 판매 4주 검증 (사전 결제 100명) → 빌드 12주

- 사전 결제 100명 즉시 모으기 어렵다고 판단
- 1인 풀타임 + RN 경험 있어 빌드 자체에 자신 있음

| Mem AI 실패 패턴 회피 장치 | 빌드인퍼블릭으로 출시 전 청중 1,000+ 확보 |

## 3. 오케스트레이션 구조

### 역할 분담

| 역할 | 담당 | 책임 |
|------|------|------|
| **의사결정자** | Ben | 큰 결정 (방향·예산·전략) |
| **오케스트레이터** | Claude | 마스터 플랜 관리, 트랙 조율, 에이전트 호출 |
| **트랙 A 서브에이전트** | general-purpose 또는 engineering | 개발 가이드·코드 스니펫·기술 문서 |
| **트랙 B 서브에이전트** | general-purpose 또는 marketing:content-creation | 마케팅 콘텐츠·카피·영상 스토리보드 |

### 주간 동기화 사이클

```
월요일 — 주간 계획 작성
  ├ Claude가 이번 주 트랙 A·B 액션 정의
  ├ 멀티 에이전트로 자산 생성 위임
  └ Ben이 검토 + 승인

수~목요일 — 중간 점검 (필요 시)
  ├ 막힌 부분 지원
  └ 트랙 간 의존성 조율

금요일 — 진행 검토
  ├ Ben이 이번 주 결과 보고
  ├ Claude가 다음 주 계획 조정
  └ KPI 업데이트
```

### 에이전트 호출 원칙

1. **큰 단위만 위임** — "이번 주 마케팅 콘텐츠 5개" 단위. "트윗 1개"는 직접 작성
2. **컨텍스트 충분히 전달** — 각 에이전트는 본 세션 모름. 페르소나·결정사항 매번 요약 전달
3. **결과 통합 후 보고** — 에이전트 raw output 그대로 X. Claude가 정리·통합
4. **병렬 호출** — 트랙 A·B 의존성 없으면 동시 실행
