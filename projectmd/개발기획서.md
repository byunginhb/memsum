# 🛠 Memsum 개발 기획서

> 하이브리드 아키텍처 (Expo + 네이티브 모듈 + WebView 일부)
> iOS 우선 출시 → Android 후속 (Month 3~4)
> 12주 빌드 + 4주 사전 판매 검증 = 16주 총 일정

---

## 1. 프로젝트 개요

### 1.1 제품 정의
**Memsum (멤섬)** — Memory + Summary
> 인스타·카톡·유튜브 스크린샷 찍는 순간, 알아서 정리하고 캘린더까지 잡아주는 iOS 외장 두뇌.

### 1.2 핵심 가치 제안 (Value Proposition)
1. **자동 캡처** — 사용자가 정리·태그·폴더 만들 필요 없음
2. **자동 캘린더** — 스크린샷의 한국어 날짜·시간 인식 → 구글 캘린더 자동 등록
3. **주간 5줄 리포트** — 매주 일요일 9시, 한 주 핵심만 압축해 푸시

### 1.3 출시 범위 (MVP)
- **플랫폼**: iOS 16+ (한국 우선)
- **언어**: 한국어 + 영어
- **국가**: 한국 App Store 우선
- **출시 시 기능 5개** (Mem AI 실패 회피 — 기능 야심병 차단)
- **출시 후 30일 핵심 KPI**: 다운로드 5,000 + Lite 결제 200명

### 1.4 출시 범위 외 (Out of Scope — V1)
- Android 버전 (Month 3~4)
- 데스크탑 앱 (Month 9)
- 사파리·크롬 확장 (Month 6)
- Apple Watch (Month 9)
- 팀 플랜 / B2B (Month 12+)
- 영문 글로벌 출시 (Month 12)

## 2. 요구사항 수집

### 2.1 기능적 요구사항 (Functional Requirements)

#### F1. 스크린샷 자동 감지
- 사용자가 스크린샷 찍는 순간 (0.5초 이내) 푸시 알림 발생
- 알림 내용: "Memsum에 저장할까요?" + 이미지 미리보기
- 1탭으로 저장 완료
- 백그라운드 동작 가능
- 사진 권한 필요

#### F2. 스크린샷 OCR + 이벤트 추출
- 한국어·영어 텍스트 인식 (정확도 90%+ 목표)
- 날짜·시간·장소·이벤트 명 자동 추출
- 한국식 날짜 표기 인식 ("다음 주 화 오후 2시", "5/15 토 14시")
- 추출 결과를 사용자에게 보여주고 [캘린더 추가] CTA 제공

#### F3. 자동 주제 분류
- 캡처된 자료를 임베딩 기반 의미 클러스터링
- 사용자가 폴더·태그 만들지 않음 (자동)
- 비슷한 과거 자료와 자동 연결

#### F4. 주간 5줄 리포트
- 매주 일요일 저녁 9시(KST) 자동 발송
- 한 주 동안 캡처한 N개 중 가장 가치 있는 5개 선정
- 각 항목: 제목 + 한 줄 핵심 + 원본 링크
- 푸시 알림 + 인앱 화면 동시 제공
- 사용자 피드백 [👍][👎]으로 다음 주 정확도 학습

#### F5. 검색 & 자료실
- 자연어 검색 ("코카콜라 캠페인")
- 결과: 캡처 카드 리스트 + 요약 + 캘린더 등록 여부
- 주제별 보기 모드 (자동 클러스터)
- Ask Your Brain (Pro 이상) — 개인 RAG, "내 자료 기반 AI 답변"

#### F6. 공유 시트 익스텐션 (iOS Share Extension)
- 인스타·사파리·트위터·유튜브에서 "공유" 버튼 → Memsum 옵션
- 1탭으로 즉시 캡처
- 미리보기 + 메모 추가 (선택)

#### F7. 위젯 & Siri 단축어
- 홈화면 위젯 — "오늘 캡처 N개" + 빠른 메모 버튼
- 잠금화면 위젯 — 작은 캡처 카운터
- Siri 단축어 — "헤이 시리, 멤섬에 저장" 음성 메모

#### F8. 구독 결제
- Free/Lite/Pro/Pro Max 4단 플랜
- Apple In-App Purchase (StoreKit 2)
- 첫 7일 무료 체험
- 사전 결제자 평생 50% 할인 코드

#### F9. 사용자 인증
- Apple로 로그인 (필수)
- 구글로 로그인 (캘린더 연동 자동 결합)
- 이메일/비밀번호 (백업)

#### F10. 데이터 백업 & 내보내기
- iCloud 자동 백업 (사용자 옵션)
- JSON / Markdown 내보내기 (Pro Max)
- 계정 삭제 시 30일 보관 후 영구 삭제

### 2.3 제약 (Constraints)
- **팀 규모**: 1인 (또는 1+외주). 풀스택 부담 큼
- **예산**: 사전 판매 ₩30만 + 본인 부트스트랩 또는 외주 ₩2,000~5,000만
- **일정**: 사전 판매 4주 + 빌드 12주 = 총 16주
- **인프라 비용 한계**: Month 1 < ₩50만 / Month 6 < ₩500만
- **개발자 경험**: React Native 학습 곡선 있음. 네이티브 모듈은 일부 외주 가능
- **App Store 정책**: Guideline 4.2 (WebView 래퍼 금지) 회피 필수

### 3.2 데이터 흐름 — 핵심 시나리오

#### 시나리오 A. 스크린샷 자동 캡처
```
1. 사용자가 스크린샷 찍음 (홈+볼륨 버튼)
   ↓
2. iOS가 사진 라이브러리에 저장
   ↓
3. PHPhotoLibraryChangeObserver가 변화 감지 (백그라운드 0.5초 이내)
   ↓
4. 네이티브 모듈이 새 이미지 메타데이터 추출
   ↓
5. 로컬 푸시 알림 발송 — "Memsum에 저장할까요?"
   ↓
6. 사용자가 알림 탭 → 앱 활성화 (혹은 알림 액션 [예])
   ↓
7. 네이티브: 이미지를 임시 캐시에 저장
   ↓
8. Vision Framework로 1차 OCR (온디바이스, 한국어·영어)
   ↓
9. 텍스트가 있으면 → 백엔드로 전송 (이미지 + OCR 텍스트)
   ↓
10. 백엔드 Edge Function:
    - GPT-4o-mini로 OCR 후처리 (오타 교정, 의미 파악)
    - 이벤트 추출 (날짜·시간·장소)
    - 임베딩 생성 → pgvector에 저장
    - 자동 주제 분류 (기존 클러스터와 유사도 비교)
   ↓
11. 응답: { captureId, eventDetected, suggestedCluster }
   ↓
12. 앱이 사용자에게 결과 표시:
    - "📦 저장 완료" (이벤트 없는 경우)
    - "📅 이벤트 감지: 6/15 (토) 14:00 코엑스 — 캘린더 추가?"
   ↓
13. 사용자가 [캘린더 추가] → Google Calendar API 호출
   ↓
14. 캘린더 등록 완료 → 캡처 메타데이터 업데이트
```

#### 시나리오 B. 일요일 저녁 9시 주간 리포트
```
1. Supabase pg_cron이 매주 일요일 21:00 KST 실행
   ↓
2. 모든 활성 사용자의 지난 7일 캡처 조회
   ↓
3. 각 사용자별 Edge Function 호출:
   - 캡처들의 임베딩 클러스터링
   - 중요도 점수 계산 (희소성 + 관심도 가중치)
   - 상위 5개 추출
   - GPT-4o-mini로 각 항목 한 줄 요약 생성
   ↓
4. 결과를 DB에 저장 (weekly_reports 테이블)
   ↓
5. Apple Push Notification 발송
   ↓
6. 사용자 탭 → WebView에서 5줄 리포트 화면 렌더
   ↓
7. 사용자 [👍][👎] → 다음 주 가중치 학습 (피드백 DB 저장)
```

## 4. 기술 스택 (요약 — 상세는 별도 문서)

### 4.1 클라이언트 (iOS 앱)
- **프레임워크**: Expo SDK 53+ (Bare workflow 가능)
- **언어**: TypeScript (RN) + Swift (네이티브 모듈)
- **상태 관리**: Zustand (간단) 또는 React Query (서버 상태)
- **네비게이션**: Expo Router (파일 기반)
- **WebView**: react-native-webview
- **알림**: Expo Notifications + APNs
- **결제**: Expo IAP (StoreKit 2 래퍼)
- **OCR (온디바이스)**: Vision Framework (Swift 네이티브 모듈)
- **이미지 처리**: expo-image, expo-image-manipulator
- **인증**: Apple Sign-in, Google Sign-in (Expo)

### 4.2 WebView 영역 (Vercel 호스팅)
- **프레임워크**: Next.js 15 (App Router)
- **UI**: shadcn/ui + Tailwind CSS
- **상태**: React Query + Zustand
- **테마**: 라벤더 보라 (#7C6FE8) — 브랜드 가이드 참조
- **빌드/배포**: Vercel (자동 배포, PR 프리뷰)

### 4.3 백엔드
- **Backend-as-a-Service**: Supabase (PostgreSQL + Auth + Storage + Edge Functions + Realtime + pg_cron)
- **Edge Functions**: Deno (TypeScript)
- **벡터 DB**: pgvector
- **AI/LLM**: OpenAI API (GPT-4o-mini for OCR 후처리·요약·분류; text-embedding-3-small for 임베딩)
- **결제**: Apple IAP (메인) + Toss Payments (사전 판매)
- **푸시**: Apple Push Notification Service (APNs)

## 5. 핵심 기술 결정 (ADR — Architecture Decision Records)

### ADR-001: Expo React Native + 네이티브 모듈 + WebView 하이브리드 채택

**컨텍스트**
- 한 벌 코드로 iOS·Android 동시 관리 + 빠른 배포 요구
- Memsum 핵심 기능 5개 중 4개(스크린샷 감지·공유 시트·위젯·OCR)는 네이티브 API 필수
- Apple App Store Guideline 4.2 (WebView 래퍼 금지) 회피 필요

**결정**
Expo (Bare Workflow 또는 Managed + Dev Client) 채택. 핵심 기능은 Expo Modules API로 네이티브 모듈 작성. 대시보드·검색·설정 등 UI 70%는 Next.js 웹앱을 WebView로 임베드.

**대안 검토**
| 대안 | 채택 안 한 이유 |
|------|----------------|
| 완전 네이티브 (Swift) | iOS·Android 코드 2벌, 1인 개발 비현실적 |
| WebView 100% (Capacitor) | 핵심 기능 4개 구현 불가, Guideline 4.2 위반 |
| Flutter | 네이티브 모듈 작성 가능하지만 한국 개발자 풀 작음, RN 생태계 대비 약함 |
| 순수 RN (Bare) | Expo 대비 빌드·배포 인프라 구축 부담 큼 |

**트레이드오프**
- ✅ 한 벌 코드 + 빠른 배포 (WebView 영역만 즉시 반영)
- ✅ Apple 심사 통과 (네이티브 비중 충분)
- ⚠️ Expo Modules API 학습 곡선
- ⚠️ WebView ↔ Native 통신(postMessage) 설계 필요

**향후 재검토 시점**
- DAU 10만 도달 시 성능 병목 확인 (WebView ↔ Native 통신 비용)
- Apple Intelligence 추격 시 네이티브 비중 더 늘릴지 결정

### ADR-002: 백엔드 = Supabase 채택 (vs Firebase / 자체 구축)

**컨텍스트**
- 1인 개발, 빠른 출시
- 한국어 임베딩·벡터 검색 필요
- 사용자별 데이터 격리 (RLS) 필수
- Apple 영수증 검증·푸시·이메일·이미지 저장 통합 필요

**결정**
Supabase 채택. PostgreSQL + pgvector + RLS + Edge Functions + Storage + Auth 일체형.

**대안 검토**
| 대안 | 채택 안 한 이유 |
|------|----------------|
| Firebase | NoSQL이 관계형 데이터에 비효율, pgvector 같은 벡터 DB 없음 |
| 자체 구축 (Node.js + PostgreSQL + Redis) | 1인 운영 부담 큼, 초기 인프라 비용·시간 ↑ |
| AWS (Lambda + DynamoDB + RDS) | 설정 복잡, 비용 추적 어려움 |
| PlanetScale + Vercel Functions | 벡터 DB 별도 필요 (Pinecone 등), 통합 부족 |

**트레이드오프**
- ✅ Auth·Storage·DB·Functions 일체형 → 빠른 개발
- ✅ pgvector 내장 → 별도 벡터 DB 불필요
- ✅ RLS로 보안 단순화
- ⚠️ Supabase 종속성 (마이그레이션 비용 크지 않음, PostgreSQL 기반)
- ⚠️ Edge Functions (Deno) 학습

**향후 재검토 시점**
- DAU 5만 + 월 비용 ₩500만 초과 시 자체 인프라 고려

### ADR-003: OCR 전략 = Vision Framework + GPT-4o-mini 2단계

**컨텍스트**
- 한국어 OCR 정확도 90%+ 목표
- iOS 기본 Vision Framework는 한국어 약함 (오타·띄어쓰기 문제)
- 비용 최적화 필요 (사용자당 월 ₩200~500 목표)

**결정**
2단계 파이프라인:
1. **1단계 (온디바이스, 빠름·무료)**: Vision Framework로 1차 OCR
2. **2단계 (서버, 정확)**: 1차 결과를 GPT-4o-mini로 후처리 — 오타 교정, 띄어쓰기, 이벤트 추출

**대안 검토**
| 대안 | 채택 안 한 이유 |
|------|----------------|
| Vision Framework 단독 | 한국어 정확도 < 70%, 시그니처 기능 신뢰도 X |
| Google ML Kit 단독 | 외부 의존성 + 네트워크 필수 |
| OpenAI Vision API 단독 | 비용 큼 (캡처당 ~$0.01), 응답 1~3초 |
| 자체 OCR 모델 | 학습 데이터·인프라 부담, 1인 비현실적 |

**트레이드오프**
- ✅ 비용 효율 (Vision 무료 + GPT-4o-mini ~$0.0001/캡처)
- ✅ 응답 빠름 (1단계 즉시, 2단계 < 1초)
- ✅ 정확도 90%+ 달성 가능
- ⚠️ 2단계 의존성 (서버 장애 시 정확도 ↓ → 1단계만 표시)

### ADR-004: 결제 = Apple IAP 메인 + Toss Payments 사전 판매

**컨텍스트**
- 사전 판매 4주 (앱 출시 전) → Apple IAP 사용 불가
- 출시 후 구독은 Apple 30% 수수료 정책 따라야 함

**결정**
- **사전 판매**: Toss Payments (간편 결제, 한국 최적)
- **출시 후 메인**: Apple In-App Purchase (StoreKit 2)
- **사전 결제자 평생 50% 할인**: 출시 시 Apple IAP에 프로모션 코드 발급

**대안 검토**
| 대안 | 채택 안 한 이유 |
|------|----------------|
| Stripe (사전 + 출시 후) | 출시 후 Apple 정책 위반, 30% 수수료 회피 시 앱 제거 위험 |
| 카카오페이 (사전) | 토스 대비 한국 IT 사용자 결제 경험 약함 |
| 자체 결제 (출시 후) | Guideline 3.1.1 위반 (구독은 IAP 필수) |

**트레이드오프**
- ✅ 사전 판매 즉시 가능 + 출시 후 Apple 정책 준수
- ⚠️ Apple 30% 수수료 (₩4,900 → 실수입 ₩3,430)
- ⚠️ Toss → Apple IAP 마이그레이션 흐름 설계 필요

### ADR-005: 데이터 프라이버시 = 사용자 디바이스 우선 + 명시적 동의

**컨텍스트**
- 사용자가 던지는 콘텐츠 = 매우 개인적 (스크린샷, 약속, 메모)
- 한국 개인정보보호법 + Apple App Tracking Transparency 준수
- AI 학습에 사용 안 함 약속이 핵심 신뢰 자산

**결정**
- 모든 캡처 데이터는 사용자 단위로 격리 (Supabase RLS)
- 이미지는 사용자 Supabase Storage 버킷에만 저장
- GPT-4o-mini 호출 시 OpenAI에 "Do not train on user data" 옵션 (API 헤더)
- 캘린더·구글 토큰은 암호화 저장 (Supabase Vault)
- 익명화된 사용 통계만 PostHog로 추적
- 계정 삭제 요청 시 30일 grace period 후 완전 삭제

**대안 검토**
| 대안 | 채택 안 한 이유 |
|------|----------------|
| 완전 온디바이스 (서버 X) | 검색·요약·RAG 기능 불가 |
| 익명 게스트 모드 기본 | 동기화·백업 불가, 디바이스 변경 시 데이터 손실 |

**트레이드오프**
- ✅ 신뢰 자산 구축 (마케팅에 활용)
- ✅ 법적 안전
- ⚠️ 익명 분석 데이터 한정으로 행동 통찰 일부 제한
- ⚠️ 운영 절차(계정 삭제 등) 인프라 필요

### ADR-006: WebView 영역 = 별도 도메인 (web.memsum.app) 운영

**컨텍스트**
- WebView가 빠른 배포의 핵심 — Vercel 자동 배포 활용
- 보안: 앱 안 WebView ↔ 외부 웹사이트 구분
- 디버깅: WebView 영역도 일반 웹사이트처럼 접근 가능해야 개발 효율

**결정**
- 메인 도메인: `memsum.com` (랜딩, 마케팅)
- WebView 전용: `web.memsum.app` (앱 내 임베드 전용, 외부 접근 시 redirect)
- 인증: JWT를 네이티브 → WebView로 postMessage 전달
- 통신: window.postMessage + 네이티브 메시지 핸들러

**트레이드오프**
- ✅ 빠른 반영 (앱 심사 없이 즉시)
- ✅ 디자인 일관성 (브랜드 가이드 그대로)
- ⚠️ WebView ↔ Native 인증 토큰 동기화 흐름 필요
- ⚠️ 오프라인 모드 한정 (서비스 워커로 부분 캐시)

- Supabase RLS 정책으로 사용자별 데이터 격리
- API Rate Limiting (Edge Functions)
- 이미지 업로드 검증 (MIME, 사이즈 제한 10MB)
- 의심 로그인 알림 (이메일)
- 보안 감사 로그 (30일 보관)

### 8.4 OpenAI 데이터 처리
- API 호출 시 헤더에 `OpenAI-DataProcessingAddendum` 설정
- 사용자 콘텐츠는 OpenAI 모델 학습에 사용 안 함 명시
- 출시 전 OpenAI 엔터프라이즈 약관 검토
